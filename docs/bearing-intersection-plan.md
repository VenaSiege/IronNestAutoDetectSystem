# 方位角交会与自动敌军坐标计算方案

## Summary

结论先说：

- 不需要把大格和小格强行融合成单视图

这个需求的核心是“几何计算 + 结果展示 + 回写标识”，不是“必须在一张图上编辑全部层级”。现有双层视图完全可以承载该需求，只需要新增一套“连续坐标 / 比例尺 / 方位角交会”的数据与计算能力，再在主视图补充结果摘要和入口即可。

建议的实现方式保留当前：

- 主视图负责全图浏览、摘要显示、入口操作
- 小格视图负责精细落点与局部检查

新增：

- 比例尺配置
- 标识点连续坐标模型
- 专用“方位角交会”计算面板
- 自动生成敌军目标点
- 自动输出“到铁巢距离”和“以铁巢为原点的方位角”

本次规格已锁定为：

- 交会结果吸附到最近小格
- 计算结果直接生成正式敌军目标点
- 输入方式使用专用计算面板

## Key Changes

### 1. 是否要改成单视图

建议不改成单视图，原因：

- 计算本身与“是否单视图”没有强依赖，几何求交在数据层完成即可
- 当前地图总规模是 `20 x 10` 大格，每大格再分 `10 x 10` 小格；如果强行全展开成单视图，本质会变成 `200 x 100` 细网格，Tkinter 下可读性和交互密度都会明显恶化
- 你后续真正需要的是：
  - 能选原点标识
  - 能输入角度
  - 能看到求交结果
  - 能回写为敌军点并输出与铁巢关系
- 这些更适合“主视图 + 侧边 / 弹窗计算面板”，而不是“把两级网格视觉上合并”

建议保留现状并增强：

- 主视图新增“计算”入口、结果摘要、必要时可显示辅助信息
- 小格视图继续负责精细定位和核对结果

如果后续确实需要“在全图上画两条射线”，那是叠加分析层，不是把大小格结构完全合并。

### 2. 数据与坐标体系

当前 `Marker` 只能表达“落在哪个小格”，还不够做稳定几何分析。需要补一套内部几何坐标体系。

建议内部统一使用连续平面坐标：

- 原点：整张地图左下角
- `x` 轴：向右为正
- `y` 轴：向上为正
- 单位：`km`

坐标换算规则：

- 设 `small_cell_km` 为 1 个小格代表的公里数
- 因为 1 大格 = 10 小格，所以 `big_cell_km = small_cell_km * 10`

若某标识位于：

- 大格列 `A-T`
- 大格行 `1-10`
- 小格 `small_x:small_y`

则其几何中心坐标为：

```text
x = ((big_col_index * 10) + small_x + 0.5) * small_cell_km
y = (((big_row - 1) * 10) + small_y + 0.5) * small_cell_km
```

本期因已选“吸附到最近小格”，所以：

- 内部仍先用连续坐标计算交点
- 最终结果再转换为“最近小格”
- 正式保存的敌军点仍沿用现有 `big_col / big_row / small_x / small_y`
- 不需要在 v1 就把 `Marker` 改成存连续坐标

但要新增辅助类型或函数：

- 标识点转几何坐标
- 几何坐标转最近小格
- 两点距离计算
- 以某点为原点的方位角计算

### 3. 比例尺配置

需要新增地图比例尺配置，建议做成全局地图设置。

建议保存字段：

- `small_cell_km`
- `big_cell_km` 可不单独存，运行时由 `small_cell_km * 10` 推导

如果坚持手工输入“大格多少 km、小格多少 km”两个值，也可以支持，但必须校验：

```text
big_cell_km == small_cell_km * 10
```

不一致时禁止保存配置并提示。

UI 方式：

- 菜单新增“地图设置”或“比例尺设置”
- 弹窗输入 1 小格代表多少 km
- 保存到 JSON 顶层配置，而不是散落在标识点里

JSON 需要升级版本，顶层建议新增：

- `version`
- `map_settings`
- `markers`

### 4. 方位角定义与交会算法

方位角定义按规则固定：

- 以原点向 `y` 轴正方向发出基准射线
- 顺时针旋转
- 与 `y` 轴正方向夹角为方位角
- 精确到小数点后 1 位

这意味着角度转方向向量时：

- 设方位角为 `theta_deg`
- 转弧度 `theta = radians(theta_deg)`
- 射线方向向量：

```text
dx = sin(theta)
dy = cos(theta)
```

因为：

- `0°` 指向正北 `(0, 1)`
- `90°` 指向正东 `(1, 0)`
- `180°` 指向正南 `(0, -1)`
- `270°` 指向正西 `(-1, 0)`

两原点交会算法：

- 已知原点 `P1(x1, y1)`、方位角 `a1`
- 已知原点 `P2(x2, y2)`、方位角 `a2`
- 射线分别为：

```text
L1 = P1 + t1 * D1
L2 = P2 + t2 * D2
```

解二维线性方程得到交点与参数 `t1, t2`。

必须处理的失败条件：

- 两条射线平行或近似平行：不给结果
- 交点落在任一射线反向上，即 `t1 < 0` 或 `t2 < 0`：不给结果
- 交点超出地图边界：提示“结果在地图范围外”，默认不写入
- 角度非法或原点不存在：不给结果

成功后：

- 先得到连续交点 `(x, y)`
- 吸附到最近小格中心
- 若该格已有标识，按现有敌军点覆盖规则处理或专门提示确认
- 自动生成 / 更新敌军目标点

### 5. 自动输出三类结果

交会成功后，系统需要自动给出三类结果：

#### 敌军坐标

- 大格坐标，例如 `F7`
- 小格坐标，例如 `3:2`

#### 敌军到铁巢距离

- 从铁巢几何中心到敌军几何中心的欧氏距离
- 单位 `km`
- 建议显示到小数点后 1 位或 2 位

#### 以铁巢为原点的方位角

- 从铁巢指向敌军
- 仍按“以正 y 轴为 0°，顺时针增加”的规则计算
- 计算方式可用：

```text
angle = degrees(atan2(dx, dy))
```

- 若结果为负，加 `360`
- 最终显示到小数点后 1 位

前置约束：

- 若铁巢不存在，则仍可算敌军坐标，但“到铁巢距离 / 方位角”不可用，并提示缺少铁巢

### 6. UI 交互方式

建议新增一个“方位角交会”专用计算面板，避免把逻辑塞进右键菜单。

建议面板内容：

- 原点 1：下拉选择 `观测点 / 参考点 / 铁巢`
- 方位角 1：输入框
- 原点 2：下拉选择 `观测点 / 参考点 / 铁巢`
- 方位角 2：输入框
- 敌军名称：文本输入
- 计算按钮
- 结果区域：显示
  - 交会结果坐标
  - 吸附后坐标
  - 到铁巢距离
  - 以铁巢为原点方位角
- 成功后自动写入正式敌军目标点

约束建议：

- 两个原点不能相同
- 原点类型允许是铁巢、观测点、参考点；不允许用敌军点当原点
- 方位角范围 `0.0 <= angle < 360.0`
- 输入显示与结果显示统一保留 1 位小数

入口位置：

- 主菜单新增“计算 -> 方位角交会”
- 或主窗口右侧增加一个折叠面板
- v1 更适合独立弹窗，改动更小

### 7. 对现有主视图 / 小视图的最小增强

不需要重做地图结构，但建议加几处增强。

主视图：

- 可在状态栏或摘要区显示“最近一次交会结果”
- 悬停敌军点时可显示：
  - 坐标
  - 名称
  - 到铁巢距离
  - 以铁巢为原点方位角
- 菜单增加“比例尺设置”和“方位角交会”

小格视图：

- 保持原有右键放点
- 交会生成的敌军点能正常显示
- 不必在 v1 画辅助射线

如果后续非常希望“直观看到两条射线”，建议作为下一阶段：

- 在主视图加一个分析叠加层
- 仅用于显示原点、射线方向、交点预览
- 仍不需要把大小格结构物理合并成一张编辑视图

## Test Plan

只做语法检查和人工验收，不自动跑测试。

人工验收场景：

1. 设置比例尺后，系统能正确显示 1 小格与 1 大格对应公里数
2. 选择两个不同原点并输入合法方位角后，能生成敌军目标点
3. 生成结果后，能看到敌军坐标、大格 / 小格位置、到铁巢距离、以铁巢为原点方位角
4. 两条射线平行或近似平行时，系统提示无法交会
5. 交点在原点反向延长线上时，系统提示无有效交会
6. 交点落在地图外时，系统提示结果越界，不自动写入
7. 铁巢不存在时，仍可求敌军坐标，但距离和铁巢方位角不给结果并提示
8. 保存并重新加载 JSON 后，比例尺配置和自动生成的敌军点都能恢复
9. 吸附到最近小格后，生成结果与地图显示一致
10. 敌军点编号仍遵守现有唯一性规则

## Assumptions

- v1 只支持两条方位角射线交会
- 比例尺以“1 小格代表多少 km”为主输入，`1 大格 = 10 小格` 由系统推导
- 自动生成的敌军点采用现有敌军目标点类型与编号机制
- 交会结果先按连续几何坐标计算，再吸附到最近小格
- 当前阶段不要求在地图上绘制射线、误差圈、历史观测层
- 不需要把大小格合并成单视图；后续若要做全图分析层，可在主视图上增量扩展
